# 项目管理文件（plan.md）建设计划

## 目标
在仓库根目录建立一个轻量的 `plan.md`，用于我们两人协作的“迭代面板”：
- 记录 **已完成** 与 **计划要做** 的需求
- 每条需求都包含：**用户故事** 与 **验收标准**
- 每次执行任务时新增一条需求卡片；验收通过后将其标记为完成
- 保持简单、可读、易维护，不引入额外工具与流程负担

## plan.md 信息架构（建议稿）
`plan.md` 采用三段式看板（便于快速扫描）：
1. **Backlog（待办）**：已明确要做，但尚未开始
2. **In Progress（进行中）**：正在执行/实现中（建议同一时间最多 1–3 条）
3. **Done（已完成）**：已通过验收的需求

每条需求卡片统一模板，确保信息完整且易回溯：
- **标题**：一句话说明做什么
- **状态**：Backlog / In Progress / Done（由所在区域隐含；也可显式写）
- **用户故事**：`作为 [角色]，我希望 [能力]，以便 [价值]`
- **验收标准**：可执行、可验证的清单（建议 3–7 条）
- **关联**（可选）：相关页面/接口/文件路径（只写关键入口，避免过细）
- **记录**（可选）：创建日期、完成日期、备注

## 维护规则（协作约定）
- **新增需求**：每次我们开始一个新任务，就在 Backlog（或直接 In Progress）新增一条卡片
- **状态流转**：执行中移动到 In Progress；验收通过后移动到 Done，并勾选验收标准
- **验收口径**：以“可复现/可验证”为准；若存在争议，以补充验收标准为优先
- **排序约定**：各栏位按“最新在上”排序，方便回看最近工作
- **粒度约定**：一条卡片对应一个可交付成果；避免把无关需求塞到同一条里

## 实施步骤（落地到仓库）
1. 检查仓库内是否已有类似规划/任务文档，避免重复建设
2. 在仓库根目录创建 `plan.md`，写入：
   - 顶部简短说明（如何使用）
   - 三段式看板（Backlog / In Progress / Done）
   - “需求卡片模板”（可复制粘贴）
3. 在 `plan.md` 里初始化 1–3 条“示例卡片”（可标注为示例，便于后续直接复用模板）
4. 之后每次执行任务：
   - 开始任务：新增卡片或将卡片移动到 In Progress
   - 交付并验收：勾选验收标准并移动到 Done
