# 产品路线图（roadmap.md）— 售前快速业务建模平台

## 0. 背景与问题定义（看方向）
我们要解决的核心问题：业务售前在客户现场需要在极短时间内完成“业务系统建模共识”，并把共识沉淀为可交付资产，进一步指导数据库生成与系统研发落地。

当前仓库已具备的关键能力（从实现反推）：
- 本体图谱建模：对象类型/关系类型/动作/数据流/规则/AI模型等可视化结构表达
- 语义查询联动：自然语言→结构化意图/字段/参数（右侧面板+底部多协议预览）
- 业务场景沙盘：基于本体穷举可执行场景 + 覆盖度与缺口（coverage & gaps）
- 资源/数据结构预测：面向“业务资源和数据结构”的推演（已有 API 与 UI 交互）

与“售前现场建模→数据库生成→研发指导”的目标差距：
- 缺少“现场交付模式”：引导式流程、时间盒、成果包一键导出、演示话术与证据链
- 缺少“工程化产出物”：数据库 DDL / ERD / 数据字典 / API 草案 / 权限模型等可落地资产
- 缺少“协作与治理”：版本、对比、审阅、签字/冻结、模板库与行业包
- 缺少“质量闸门”：命名规范、唯一性/基数/外键校验、可追溯映射（本体→表/字段→接口）

产品定位（一句话）：
- 面向业务售前的“现场业务建模工作台”：用可视化本体+LLM 推演，把客户语言快速固化为可执行的业务模型与工程资产包。

目标用户与使用场景：
- 售前顾问：现场梳理对象/流程/规则，做共识与方案证明
- 解决方案架构师：把模型转成数据库与接口草案，形成研发输入
- 客户关键用户/业务负责人：确认范围、流程、口径与边界条件
- 交付/研发负责人：接收可追溯的模型与资产包，降低返工

价值主张（对外）：
- 2 小时内完成“业务对象+关系+关键动作+规则+数据字典”的可评审版本
- 现场即可演示“语义查询→结构化→多协议预览→模拟回执”，增强可信度
- 会后 1 天内输出“数据库与接口草案”，指导研发拆解

北极星指标（North Star）：
- 从 0 到“可交付资产包”的平均耗时（TTV）
- 单次售前建模会的模型覆盖度（覆盖的 Action/Object/Link/Rule 占比）
- 交付后返工率（因口径不清导致的表/接口重做比例）
- 资产复用率（模板/行业包在不同客户的复用次数）

非目标（明确不做，避免发散）：
- 不在近期做完整的低代码/代码生成 IDE
- 不替代客户现有主数据治理平台；我们只输出可落地的设计输入与草案


## 1. 产品目标（定目标）
### 1.1 近期（售前可用的 MVP）
目标：让售前在客户现场“可引导、可产出、可导出、可复盘”。
必须交付的成果包：
- 业务模型：ObjectType/LinkType/ActionType/BusinessRule 的结构化定义
- 数据字典：字段、类型、必填、枚举、口径说明
- 场景覆盖：可执行场景清单 + 覆盖度/缺口
- 工程草案：数据库 DDL（至少 PostgreSQL/MySQL 二选一）+ ERD（可选）

### 1.2 中期（工程落地可用）
目标：把“模型→工程资产”变成稳定流水线，可指导研发拆分。
必须交付的成果包：
- 数据库：DDL + 迁移策略（增量变更/版本）
- API：GraphQL/REST 草案（与现有“模拟 GraphQL”能力对齐）
- 权限：角色/数据权限维度与约束（最小可用版）

### 1.3 远期（平台化与行业化）
目标：行业模板库+协作治理+可规模化交付。
- 行业包：ERP/供应链/制造/政务等领域模板
- 协作：多人编辑、评审、冻结、差异对比、审计日志
- 生态：对接真实 DB/代码仓库/需求系统，形成闭环


## 2. 路径拆解（拆路径）
按“现场售前建模→工程化输出→研发指导”的链路拆分能力域：

### A. 现场建模工作台（体验链路）
- A1 引导式建模：从“业务目标→角色→对象→动作→规则→数据”逐步推进
- A2 时间盒模式：40/60/90 分钟会议模板（议程+产出检查点）
- A3 证据链：每个模型元素必须可追溯到“客户原话/文档片段/会议纪要”
- A4 一键成果包导出：Markdown/PDF（可后置）+ JSON（工程输入）

### B. 模型质量与可执行性（质量闸门）
- B1 命名与规范：apiName 唯一、字段命名、类型映射、必填/枚举
- B2 关系与约束：基数、外键、逆向关系、删除策略（软删/级联）
- B3 覆盖度与缺口：延续现有“场景沙盘”的 coverage/gaps 思路，做成仪表盘
- B4 可执行演示：语义查询→预览→模拟回执（售前关键展示闭环）

### C. 工程化产出（数据库/接口/规则）
- C1 DB 生成：ObjectType/Property/LinkType → 表/字段/索引/外键（DDL）
- C2 数据字典：从模型自动生成，支持导出与客户签字确认
- C3 API 草案：ActionType → mutation/query；ObjectType → type；参数 → input
- C4 规则落地：BusinessRule → 约束（DB/应用/规则引擎）建议与占位实现

### D. 模板与行业化（复用）
- D1 行业模板：以“示例场景”机制演进成可配置行业包
- D2 组件化资产：对象/流程/规则可复用、可组合、可替换
- D3 方案库：把“演示脚本+场景沙盘+语义查询样例”打包成售前套件


## 3. 里程碑规划（可按迭代执行）
### M0（当前基线）
- 已具备：可视化本体建模、语义查询联动、多协议预览、业务场景沙盘、资源预测交互雏形

### M1（售前 MVP：现场可用）
- 引导式建模流程（最小可用）：会议模式入口 + 步骤导航 + 检查点
- 一键导出成果包：模型 JSON + 数据字典 Markdown + 场景沙盘结果
- 模型质量闸门 v1：命名唯一/必填/类型合法性/外键一致性
验收：在一个真实客户会议中完成建模并导出成果包；客户可读、研发可接

### M2（数据库生成：工程可用）
- 从模型生成 DDL（MySQL 或 Postgres 先做一个）
- LinkType→外键与关联表策略（1-1/1-n/n-n）
- 变更管理 v1：版本号 + diff（至少导出差异报告）
验收：用导出的 DDL 在本地建库成功；字段与关系与模型一致

### M3（研发指导：接口草案与拆解）
- ActionType→API 草案生成（GraphQL 优先，与现有模拟对齐）
- 数据字典增强：口径、枚举、示例值、校验规则
- 交付包增强：DB+API+规则建议文档一键打包
验收：研发可据此拆任务并进入实现，无需二次“翻译会议”

### M4（行业化与协作）
- 行业模板库（至少 2 个行业包）
- 多人协作/评审/冻结
- 审计与可追溯（谁改了什么、为什么）
验收：跨客户复用模板，显著降低建模时间（TTV 降低 50%）


## 4. 资源配置（配资源）
团队角色（最小编制）：
- 产品经理（你/我）：会议流程、交付包定义、验收口径、行业模板策略
- 全栈工程（1-2）：建模引导 UI、导出/版本、DB/API 生成器
- 语义/LLM 工程（1）：语义解析提示词与结构化输出质量、可追溯证据链
- 领域专家（兼职）：行业对象/流程/规则模板评审
- 设计/UX（兼职）：售前现场模式的信息层级与可演示性
- QA（兼职）：导出正确性、DDL 可执行性、回归

关键依赖与成本项：
- 大模型供应与稳定性（提示词、速率、成本、离线策略）
- 模型与导出格式的“版本兼容”策略（避免资产失效）
- 行业模板的版权与沉淀机制（内部资产化）

风险与应对：
- 现场信息不完整 → 用“缺口清单 + 待确认问题”输出，纳入交付包
- DB 生成争议（范式/性能/习惯）→ 提供可配置策略与默认推荐
- 模型膨胀 → 强制“会议时间盒 + 最小可交付范围”


## 5. 下一步（落到迭代）
建议下一迭代（Sprint）只抓 3 件事：
1) 售前现场模式入口 + 引导式步骤（A1/A2）
2) 一键导出成果包 v1（A4 + C2）
3) DB 生成器 v0（C1，先支持单库/基础外键）
