# 产品路线图 v2（roadmapv2.md）— 系统应用的数据 Meta Core + ORM Layer

## 0. 一句话定位
把系统应用的“数据与业务语义”先做成可计算的 Meta（元信息/元模型），再把这套 Meta 映射成数据库、接口与校验规则等工程资产：它既是数据 Meta 信息化构建的核心层，也是面向落地的 ORM 层。

## 1. 核心理念（重新规划的总纲）
### 1.1 两层内核
- Meta Core（元信息核心层）：用结构化、可演化、可推理的方式描述业务系统的“语义与数据形态”。
- ORM Layer（映射/落地层）：把 Meta Core 稳定、可追溯地转换为工程可执行资产（DB/API/规则/编排），并支持验证与演示。

### 1.2 关键原则
- Meta-First：所有工程资产都从 Meta 派生，避免“口头需求 → 手写表/接口”的二次翻译。
- One Source of Truth：Object/Link/Action/Rule/DataFlow 等统一为同一套元模型，不分散在不同文档与工具。
- Traceable by Design：每个落地资产都能追溯回 Meta 定义与变更（未来可扩展到证据链）。
- Verifiable Outputs：导出物必须可验证（DDL 可执行、Schema 可校验、模拟回执可复现）。

## 2. 将平台现有概念映射到新分层
### 2.1 Meta Core 对应概念（“描述业务语义”）
- ObjectType：领域对象/实体类型（将成为 ORM 的实体/表来源）
- LinkType：对象间关系（将成为外键/关联表/导航属性来源）
- ActionType：可执行动作/用例入口（将成为 API/Mutation/Command 来源）
- DataFlow：动作与步骤编排/依赖（将成为工作流/编排建议来源）
- BusinessRule：约束与规则（将成为 DB 约束/应用校验/流程门禁来源）
- AIModel：智能能力声明（将成为推演/生成/校验的可插拔能力来源）
- AnalysisInsight：分析结论与模型质量洞察（覆盖度、缺口、风险项）

### 2.2 ORM Layer 对应能力（“把语义变成可执行资产”）
- DB 资产：表/字段/索引/外键/中间表策略、迁移脚本、数据字典
- API 资产：GraphQL/REST schema 草案、请求/响应结构、参数校验点
- 规则落地：约束分类与实现占位（DB/应用/流程），导出与回归校验
- 运行沙盒：基于导出的契约进行模拟调用与回执（receipt）复现

### 2.3 现有 UI/能力在 v2 中的角色
- 图谱画布：Meta Core 的可视化阅读与导航入口（Object/Link 为主骨架）
- 右侧面板：Meta 元素的详情查看/编辑器（属性、规则、动作、关系）
- 语义查询：自然语言 → Meta 查询/动作意图 → 结构化模板（并联动预览）
- 多协议预览：将 Meta 派生为 RDF/OWL/SWRL/DSL/GraphQL 模板等“可计算输出”
- 模拟回执：把“预览的契约”落到可执行的模拟调用闭环，作为可演示的验证点
- 业务场景沙盘：以覆盖度/缺口视角衡量 Meta Core 的完整性与可执行性

## 3. 能力域拆解（从 Meta 到工程落地的链路）
### A. Meta Core 建模与治理
- 元模型一致性：类型系统、命名规范、引用完整性、基数与逆向关系一致性
- 模块化：按业务域/示例场景组织（模块可切换、可导出、可复用）
- 版本化：变更记录、diff、冻结点（为 ORM 导出提供稳定基线）

### B. Meta → 资产派生（ORM Layer）
- 映射规则：Object/Property/Link → DB（表/列/外键/关联表）与类型映射策略
- 契约生成：Action/DataFlow/Rule → API 草案 + 编排建议 + 校验点清单
- 产物打包：模型 JSON + 数据字典 + DDL + API schema + 规则建议文档

### C. 可验证与可演示（闭环）
- 合法性校验：导出前的质量闸门（缺失映射、类型不兼容、基数冲突等）
- 模拟运行：对生成的 API 契约进行模拟调用，输出可复现 receipt
- 测试化：导出最小回归用例（契约校验、规则断言、示例数据生成）

### D. 行业化与生态
- 行业包：以“示例场景”演进为可交付模板（对象/动作/规则/映射策略）
- 集成：对接真实数据库/代码仓库/需求系统，形成“Meta → 工程”流水线
- 协作：评审、权限、审计（围绕 Meta 与导出物的治理）

## 4. 里程碑（v2 路线图）
### v2-M0（基线：现有能力）
- 现状：可视化本体建模、语义查询联动、多协议预览、业务场景沙盘、模拟回执链路雏形
- 价值：证明“Meta 可计算”与“演示闭环”可行

#### v2-M0 基线定义（当前仓库已具备）
- 可视化本体建模：ObjectType/LinkType/ActionType/DataFlow/BusinessRule/AIModel/AnalysisInsight 的结构化表达与查看
- 语义查询联动：自然语言 → 结构化解析结果，并联动预览与右侧面板
- 多协议预览：RDF/OWL/SWRL/DSL/GraphQL 模板等可计算输出
- 业务场景沙盘：基于当前本体 MECE 穷举场景 + coverage/gaps
- 模拟回执雏形：对 GraphQL 模板进行模拟执行，返回可展示的 receipt

#### 演示闭环脚本（Meta 可计算 + receipt 可复现）
- Step 1：在顶部栏点击「加载示例」选择业务域（图书馆 / ERP）
- Step 2：点击「语义查询」，输入一条业务语句（建议用对应示例域的核心动作）
- Step 3：等待解析完成后，在预览面板查看 GraphQL 模板（GraphQL 页签）
- Step 4：点击「执行」，得到模拟回执（receiptId / status / executedAt / latencyMs）并展示返回数据
- Step 5：切换到另一业务域重复 Step 2-4，用同样闭环证明“同一元模型框架可跨域复用”

#### v2-M0 基线缺口（将由 M1/M2 补齐）
- 缺少统一的 Meta Core 质量闸门（命名/引用/基数/类型合法性）与一键校验入口
- 缺少模块化导入/导出与版本 diff（可读变更报告、冻结基线）
- 缺少从 Meta → DB/API 的工程化落地资产（DDL/数据字典/映射编辑器）

### v2-M1（Meta Core 固化：统一语义内核）
- 交付物：
  - Meta Core 质量闸门 v1（命名/引用/基数/类型合法性）
  - Meta 模块化导入/导出 v1（按业务域打包）
  - 版本与 diff v1（可生成变更报告）
- 验收口径：
  - 能在不同示例场景间切换且保持元模型一致
  - 对同一 Meta 变更能生成可读 diff，并可作为导出基线

### v2-M2（ORM Layer v1：数据库与数据字典落地）
- 交付物：
  - ObjectType/Property/LinkType → DB DDL v1（至少支持一种方言）
  - 数据字典导出 v1（字段口径、类型、必填、枚举、关系说明）
  - 映射编辑器 v1（可配置表名/字段名/主键策略/关系策略）
- 验收口径：
  - 导出的 DDL 可在本地建库成功，结构与 Meta 一致
  - 映射缺失/冲突有明确错误提示与定位

### v2-M3（契约与规则：Action/DataFlow/Rule → API + 校验点）
- 交付物：
  - ActionType → API 草案生成 v1（GraphQL 优先，与现有预览/模拟对齐）
  - BusinessRule 分类与导出 v1（DB/应用/流程三类）
  - DataFlow → 编排建议 v1（步骤与依赖的可导出表示）
- 验收口径：
  - 同一动作能同时导出 API schema 与校验点清单
  - 语义查询产出的模板可直接落到模拟调用并得到可复现 receipt

### v2-M4（工程化流水线：可复用交付包与集成）
- 交付物：
  - 一键交付包 v1（DB + API + 规则建议 + 变更报告）
  - 行业包机制 v1（模板 + 映射策略 + 演示脚本）
  - 外部集成占位（代码生成/迁移工具/仓库对接的接口边界）
- 验收口径：
  - 对同一行业包能在新项目中快速复用并导出一致资产
  - 交付包可用于研发拆解，无需二次“翻译会议”

## 5. 统一产物清单（面向验收/交付）
- Meta 产物：模型 JSON、变更 diff、模块包（可复用）
- DB 产物：DDL、迁移建议、数据字典、关系说明
- API 产物：Schema 草案、请求/响应结构、参数与校验点
- 规则产物：规则分类、落地建议、断言/测试占位
- 演示产物：模拟调用脚本/结果、receipt、关键场景清单与覆盖度

## 6. 非目标（避免发散）
- 不把自己做成完整 IDE 或全自动代码生成器；优先把“Meta → 可验证工程草案”做对
- 不替代客户现有数据治理平台；聚焦系统应用“语义与结构”的元信息层与映射层
